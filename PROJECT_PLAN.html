Two-Repo Architecture

Private — never published
btc-quantum-wiki

llm-wiki v0.3.0 instance (MIT licensed, NVK's tool). All source documents ingested here. Compiled knowledge base. You query this when writing artifacts. The engine room.

Public — GitHub Pages
btc-quantum-readiness

Public-facing site at quantum.bitcoinbutlers.com. All artifacts, thesis, explainers. Version history via Git. PRs accepted at maintainer's discretion. The public face.

FIREWALL RULE: The coding agent has write access to the public repo only. The private wiki is read-only for reference. These repos are never merged. Content derived from the private wiki is rewritten in the maintainer's voice for the public repo — wiki output is not copied directly into public files under any circumstances.

Governing Rules — Non-Negotiable

Rule 1 — No shortcuts

Every claim in every public artifact traces to a primary source verified directly. No citing secondary coverage as the source. If the primary source cannot be found and verified, the claim does not appear. No exceptions without explicit maintainer approval.

Rule 2 — Translation without condescension

Written for a CFO who has never read a BIP and a retail holder who has never read a cryptography paper. Plain language first. Technical terms defined inline on first use. Succinct, not comprehensive. If a sentence requires a cryptography background to parse, rewrite it.

Rule 3 — Maintainer is the arbiter

PRs are welcome. The maintainer accepts, rejects, or modifies at their discretion. No formal process, no timelines. Version history is public. Contributors who disagree with editorial decisions can fork.

Rule 4 — Opinions diverge here explicitly

Where this project's position diverges from NVK's or other researchers, both positions are stated with attribution. Neither is suppressed. The divergence is part of the value.

Rule 5 — Animations deferred

No animation work until core documents and thesis are complete, accurate, and reviewed by the maintainer. Animations serve translation — not decoration.

Rule 6 — Funding has no editorial rights

Sponsors acknowledged in README. No special treatment in how their work is characterised. If their paper scores low in the maintainer's private assessment, the public artifact reflects that honestly.

Rule 7 — Private until Strategy outcome or deadline

No community outreach, no Delving Bitcoin submission, no contact with NVK, no HRF approach until Strategy has reviewed the direction OR 2026-06-01, whichever comes first. This project has independent standalone value — if Strategy declines the role, it launches on the deadline regardless. The gate is the Strategy process window, not a permanent veto.

Document Artifact Template

Every public document follows this structure exactly. Query the private wiki before writing each section. Verify every claim against its primary source directly — not against the wiki's synthesis.

## [Full Title] **Authors:** [Name (Affiliation) · Name (Affiliation)] **Published:** [Date] **Type:** [BIP / Academic Paper / Industry Analysis] **Status:** [Draft / Active / Superseded] **Primary source:** [Direct URL — verified, not secondary coverage] --- ### What this is [2–3 sentences. Plain language. What problem does this address and why does it matter to someone who holds Bitcoin? No jargon without inline definition. Written for a CFO or retail holder.] ### The core argument [Accurate technical summary. No editorialising. Contested claims noted as contested.] ### Key claims - [Claim] — [[Author, year](direct url)] (add secondary corroborating primary source only if triangulation is needed) ### Where this project agrees [Alignment with our thesis — brief, specific] ### Where this project diverges [Where our editorial position differs — stated plainly, attributed. If no divergence, write: "No material divergence at this time."] ### Open questions [What this document explicitly does not resolve — honest gaps only] ### References [Primary sources only. No secondary coverage.] --- *Last updated: [date] · [contributor]* *Corrections welcome via pull request.*

Thesis Structure

Sections 1–4 are neutral synthesis — open to correction via PR. Section 5 is the maintainer's editorial position — attributed, clearly labelled, raised as comments not PRs. A rough notes file for section 5 must be started in Phase 0 and updated throughout all phases. It is not written cold in Phase 4.

# Bitcoin Quantum Readiness — Working Thesis *Sections 1–4: neutral synthesis. Open to correction via PR. Section 5: editorial position of this project. Attributed. Where this project diverges from other researchers, both views are stated with attribution.* --- ## 1. The Problem What quantum computers can do to Bitcoin's signature scheme specifically. What they cannot do — SHA-256, proof of work, hashed addresses pre-spend. Primary source citations only. No secondary coverage as source. ## 2. The Exposure How much Bitcoin is actually at risk and why. Address type breakdown — P2PK, P2TR, reused addresses, hashed pre-spend. Harvest-now-crack-later as the near-term threat. Fast-clock architectures: even briefly exposed keys at risk. Verified numbers: Blockstream breakdown, Google 2026 paper, Presidio paper. ECDLP challenges paper (arxiv 2508.14011) cited as supporting reference for attack threshold claims. ## 3. The Timeline Verified hardware data points only — no projections without citation. Today: ~100 logical qubits. IBM Starling 2029 target: 200 logical (roadmap, not achieved). Attack threshold: lower bound ~523 logical qubits per cited academic estimate; upper bound not pinned to single source (literature ranges widely). Google 2026 ~500k physical qubits as concrete alternate anchor. Governance migration timeline against hardware timeline. Where those lines may cross — uncertainty bands explicit throughout. ## 4. The Options — Comparatively Threat-Modelled BIP-360: clean soft fork, new output type, no existing coins touched. BIP-361: hard-fork-like consequences despite soft fork structure. — Phase A / B / C explained in plain language. — Community opposition documented and attributed. QSB / Binohash: no-fork bridge, works today. — Built by BIP-360's own co-author. This provenance matters. Blockstream / Liquid: SHRINCS / SHRIMPS in production. — NIST ML-DSA and FALCON standardised August 2024. ## 5. The Recommendation *Editorial position of this project. Not the views of funders, contributors, or affiliated organisations.* What institutions holding Bitcoin at scale should do now. What the governance conversation needs to look like. What the decision timeline genuinely is. Where this project diverges from other researchers — attributed, not suppressed.

Devil's Advocate — All Points Resolved

Point 1 — Resolved in v1.0
Phase 0 depends on Claude Code — not available to coding agent

llm-wiki runs as a Claude Code plugin. The ingest commands are Claude Code slash commands, not bash commands. A coding agent without Claude Code cannot execute Phase 0.

Resolution: Phase 0 is explicitly labelled as HUMAN · Claude Code. The coding agent starts at Phase 1 only after the maintainer confirms Phase 0 is complete and the wiki is queryable. The two phases are separated by an explicit checkpoint.
Point 2 — Resolved in v1.0
No documented firewall between private and public repos

Without an explicit firewall, a coding agent handed both repos could accidentally commit wiki content to the public repo, contaminating the separation.

Resolution: Firewall rule is stated at the top of the architecture section and in Phase 0. The coding agent has write access to the public repo only. The private wiki is read-only for reference. Wiki output is never copied directly to public files — it is rewritten in the maintainer's voice.
Point 3 — Resolved in v1.0
Ingesting NVK's site creates a citation loop risk

NVK's bitcoinquantum.space synthesises primary sources — it is already secondary. Treating the wiki's synthesis of his synthesis as a source would remove the primary source layer entirely.

Resolution: Explicit note in Phase 0 ingest instructions: bitcoinquantum.space is ingested for editorial awareness only. Every factual claim that originates from his synthesis must be verified one level further against the original paper he references. The wiki surfaces his framing; the maintainer traces every claim to its primary source.
Point 4 — Resolved in v1.0
Section 5 written cold in Phase 4 risks disconnection from evidence

Starting section 5 only after all artifacts are done means the editorial position hasn't been shaping the work throughout — it gets bolted on at the end.

Resolution: section-5-notes.md starts in Phase 0 and is updated at the end of every phase. By Phase 4, the notes reflect a substantive, evidence-grounded position. The coding agent does not write section 5 — it formats and commits the maintainer's content.
Point 5 — Resolved in v1.0
hardware-timeline.json required decisions the plan hadn't made

A coding agent cannot determine which data points belong in the JSON or which sources to cite — it would have guessed or hallucinated.

Resolution: The exact data points, exact values, and exact source URLs are specified in the plan above. The coding agent writes the JSON from the specification, not from its own knowledge.
Point 6 — Resolved in v1.1
Medium blog post cited as primary source for the 2500-qubit attack threshold

A Medium post was cited as one of the two anchor sources for the attack threshold range. This violates Rule 1 (no secondary coverage as primary source). An agent following the spec would have shipped a rule violation on day one; an agent flagging it would have stalled Phase 1.

Resolution: Medium source removed from hardware-timeline.json. Attack threshold now has a lower bound from the cited academic preprint and the Google 2026 physical-qubit estimate as the concrete alternate anchor. Upper bound in logical qubits is intentionally not pinned to a single source — the plan's note explains the literature range.
Point 7 — Resolved in v1.1
Starkware QSB primary source was the news article, not the paper

The Document Index listed "news.bitcoin.com QSB article (Levy's paper)" — a contradiction. The news article is coverage; Levy's paper is the primary source. The agent would have had no canonical paper URL to verify against.

Resolution: Phase 0 now instructs the maintainer to locate and pin Levy's paper and/or Starkware's canonical repo as the primary source before Phase 3. The news article is explicitly downgraded to framing-only status, alongside bitcoinquantum.space. Document Index updated accordingly.
Point 8 — Resolved in v1.1
No URL pre-flight gate — ingest could fail silently mid-execution

If any ingest URL 404s, the original plan had no gate to catch it before commands ran. The maintainer could have ingested a partial corpus and proceeded to Phase 1 with a gap.

Resolution: Step 0.3 adds an explicit pre-flight script that curls every URL and requires every status code to be 200 before ingest begins. If any URL fails, the plan is bumped to v1.2 with the updated URL before re-dispatch.
Point 9 — Resolved in v1.1
arxiv 2508.14011 appeared in ingest list with no labelled role

A paper in the ingest list with no corresponding artifact in the Document Index would have left the agent uncertain whether to write about it or treat it as supporting material.

Resolution: Paper labelled as "ECDLP challenges paper — supporting reference only, no standalone artifact" in the Phase 0 ingest list. Thesis section 2 cites it explicitly as supporting reference.
Point 10 — Resolved in v1.1
No contingency if Strategy doesn't greenlight

Every Phase 7 activity was gated on a single external decision. If Strategy passed on the maintainer, the project had no fork — public launch could be deferred indefinitely.

Resolution: Rule 7 now reads "until Strategy outcome OR 2026-06-01, whichever first." Project launches independently on the deadline regardless of Strategy outcome. The gate is the interview window, not a permanent veto.
Point 11 — Resolved in v1.1
llm-wiki commands may have drifted from plan

NVK's tool is actively developed. The original plan used /install-plugin github:nvk/llm-wiki from his X post and referenced /wiki:thesis, which no longer exists in v0.3.0 (folded into /wiki:research --mode thesis).

Resolution: Plan pins llm-wiki v0.3.0 explicitly. Phase 0 Step 0.1 instructs the maintainer to verify the current install command against the README, with marketplace-install as the likely path and manual-sync as fallback. /wiki:thesis is explicitly forbidden.
